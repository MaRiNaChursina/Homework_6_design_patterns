## Отчёт по найденным проблемам и исправлениям

Задача была — привести проект в более нормальный вид исправив в коде проблемы поддерживаемости. По смыслу программа осталась той же: вводим сумму в USD и получаем конвертацию в несколько валют по курсам из API.

---

## Проблемы и исправления

### 1) Странная "архитектура" с кучей классов
Было 4 класса: `UsdRubConverter`, `UsdEurConverter`, `UsdGbpConverter`, `UsdCnyConverter`. По факту они делали одно и то же, но:
- каждый по‑разному ходил в API;
- в каждом были методы для всех валют, а “не свои” методы просто печатали `This is not ...`.

Почему это плохо:
- добавление новой валюты = копировать ещё один файл + менять методы;
- много дублирования;
- по поведению классы ломают ожидания (метод есть, но он не конвертирует, а просто печатает сообщение).

Что сделала:
- сделала один нормальный конвертер `CurrencyConverter` с методом `convert(amount, target_currency)`;
- отдельно вынесла получение курсов в `ExchangeRateApiProvider` (одна ответственность — один класс).

---

### 2) DRY / KISS / YAGNI
**DRY**: было 4 варианта `get_rates()` и куча повторяющихся кусочков кода.  
Сделала один источник: `ExchangeRateApiProvider.get_rates()`.

**KISS**: Я оставила простую схему: один запрос с таймаутом + нормальная ошибка `ExchangeRatesError`, если API не ответило или ответило странно.

**YAGNI**: было много того, чем программа реально не пользуется (например, `asyncio` в `main.py`). Я это убрала, чтобы код был короче и понятнее.

---

### 3) SOLID

3.1 старый `CurrencyConverter(ABC)` заставлял каждый класс реализовать методы для всех валют, даже если класс "про одну валюту".  
Теперь интерфейс нормальный: один метод `convert(amount, target_currency)`.

3.2 раньше объект формально подходил по типу, но часть методов "не работала" (просто `print`). Это ломает заменяемость.  
Теперь либо конвертация возвращает число, либо кидает осмысленную ошибку.

3.3 конвертеры смешивали сразу всё: сеть, кеш/ретраи/логирование и конвертацию.  
Сейчас разделено: провайдер курсов отдельно, конвертация отдельно.

3.4 раньше конвертеры напрямую зависели от `requests.get(...)`.  
Сейчас `CurrencyConverter` зависит от абстракции `ExchangeRatesProvider`, а конкретный провайдер (`ExchangeRateApiProvider`) подкладывается снаружи.

---

### 4) "Чистый код"

- В `main.py` убрала 4 почти одинаковых блока и сделала список валют `targets` + цикл.
- Сделала более нормальный ввод суммы: теперь можно вводить дробные значения и запятую (`10,5`), плюс есть проверки на отрицательные значения.
